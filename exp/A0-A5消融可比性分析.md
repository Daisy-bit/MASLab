# A0–A5 消融可比性分析（mad_scc / soo_scc）

## Q1：任务自适应 prompt 在 A1–A5 中是否不变？

**完全不变，是受控因素。**

依据是 yaml 配置 diff —— 把 `mad_scc/configs/` 和 `soo_scc/configs/` 各自的 6 个 yaml 全部比对过，在每个方法**内部**，A0–A5 之间**只有 3–4 行 `enable_*` 开关在变**，其他全部一致。

### mad_scc 一侧

prompt 不来自 yaml，而来自 `mad_vote/prompts.py`（继承自 `MAD_Vote_Main`）。这套 prompt 只分 mcq vs 非-mcq 两档，且 A0–A5 都走同一份模板。所以 **mad_scc 内部 prompt 是常量**，但它本身并不是"任务自适应"（不区分 math/open，没有 few-shot）。

### soo_scc 一侧

prompt 来自 `soo_centered_v3_main.py:180-257` 的 `_init_prompt` / `_update_prompt` / `_update_prompt_leader_agent`，是真正的三档任务自适应版本。控制它行为的三个 yaml 字段：

| 字段 | A0 | A1 | A2 | A3 | A4 | A5 |
|---|---|---|---|---|---|---|
| `include_math_few_shot` | true | true | true | true | true | true |
| `math_mode` | complex | complex | complex | complex | complex | complex |
| `include_mcq_format_hint` | true | true | true | true | true | true |
| `force_task_type` | null | null | null | null | null | null |

**6 个变体完全一致**。所以 soo_scc 内部，包括 8 道 DyLAN-Math few-shot block 在内的整套任务自适应 prompt 也是常量。

---

## Q2：A0–A5 在每个方法内部是否可比？

**是。** 两个方法各自的 A0–A5 都是严格控制变量的设计 —— 只有核心 SCC 组件在变，所有其他基础条件（prompt、role personas、embedding 模型、judge、agent 数、轮数、随机种子、数据集、温度、shared initial pool）都被钉死。

---

## A. mad_scc 内部演化（A0 → A5）

### 跨 A0–A5 严格相等的基础条件

| 项 | 值 | 来源 |
|---|---|---|
| 父类 | `MAD_Vote_Main` | `mad_scc_main.py:120` |
| Round-0 prompt | `get_initial_user_prompt(query, task_type)`（mcq 一档/其他一档） | `mad_vote/prompts.py:59-71` |
| Round-update prompt | `get_debate_user_prompt(query, peer_responses, task_type)` | `mad_vote/prompts.py:74-96` |
| 5 个 agent 系统 prompt | Algebraic-Reasoner / Numerical-Verifier / Concrete-Example-Reasoner / Structural-Decomposer / Pragmatic-Heuristic-Solver | `mad_vote/prompts.py:10-56` |
| `agents_num` / `rounds_num` | 5 / 3 | yaml |
| Round-0 答案 | 从 `POOL_DIR` 回放（5 个变体读同一份缓存） | `run_mad_scc_1.5b.sh:59,152-153` |
| `task_type` | `auto`（同一个 dataset 解析结果） | yaml |
| Judge | xverify-9b-c，cached on canonical | yaml + base class |
| Equivalence 协议 | `task_aware` | yaml |
| Top-k / sim_threshold / diversity_p / enforce_dag | 2 / 0.75 / 0.0 / true（仅 routing 开时生效） | yaml |
| Embedding 模型 | `model/all-MiniLM-L6-v2` | yaml |
| variance_consensus_thr / answer_consensus_min | 0.05 / 3（仅 triggering 开时生效） | yaml |

### 仅有的差异：三个 `enable_*` 开关

| Variant | `enable_triggering` | `enable_routing` | `enable_weighted_aggregation` |
|---|---|---|---|
| A0_vanilla | false | false | false |
| A1_triggering | **true** | false | false |
| A2_routing | false | **true** | false |
| A3_aggregation | false | false | **true** |
| A4_all | true | true | true |
| A5_trig_rout | true | true | false |

### A0_vanilla：行为锚点

代码路径（`mad_scc_main.py:388-687`）：

1. **Spectral 分析**：`need_spectral = False` 短路（`:471-475`），不调 embedding，`contributions = [1/N, ..., 1/N]` 均匀。
2. **Vote**：`_vote` 走 `enable_weighted_aggregation=False` 分支（`:283-288`），打分 `(size, -groups.index(g))` —— 纯多数投票 + 首组 tiebreak。
3. **Triggering**：`_trigger_hit` 第一行 `if not self.enable_triggering: return False, None`（`:301-302`）—— 直接跳过早停。
4. **Routing**：`_peers_per_agent` 第一行 `if not self.enable_routing: return [[j for j in range(n_agents) if j != i] for i in range(n_agents)]`（`:358-359`）—— 全连接 mesh。
5. **结果**：纯 mad_vote vanilla MAD —— 5 个 agent 在 round 0 各自独立作答，round 1/2/3 每轮并发广播看上轮快照，3 轮跑满，最终等权 plurality。

### A1_triggering：仅打开早停

差分（vs A0）：

- `need_spectral` 变 `True`，每轮都跑 `pc1_contributions(embs)`，得到 PC1 softmax 贡献分（`:476-483`）。
- `_trigger_hit` 实际执行（`:298-341`）：
  - 闸门 a：`count_first_plurality` 命中 task-aware 等价类大小 ≥ 3 → `"answer_plurality"`
  - 闸门 b：`is_spectral_consensus(spec_diag, 0.05)`（tr(S_c) < 0.05）→ `"spectral_trace"`
  - 任一命中 → `early_exit_round = r`
- 后续轮在 `:512-524` padding：不调 LLM，沿用上轮 vote / extracted / correct 历史。

**未变**：vote 仍然走等权 plurality（`enable_weighted_aggregation=False`），peer 仍然全连接（`enable_routing=False`）。contributions 算出来但只参与早停判断，**不参与投票**。

**结果**：可能在 round 0 / 1 / 2 直接收敛，也可能跑满 3 轮；投票规则和拓扑与 A0 完全一致。这就是隔离"早停"模块的纯效应。

### A2_routing：仅打开 DAG 路由

差分（vs A0）：

- `need_spectral = True`（routing 需要 contributions 算图）。
- `_peers_per_agent` 走 `:361-383`：
  1. `pairwise_cosine(embs)` → `sims`
  2. `build_diverse_graph(sims, contributions, n, top_k=2, sim_threshold=0.75, diversity_p=0.0, enforce_dag=true)` 算出 directed DAG `edges`
  3. `topo_order_by_contributions(edges, contributions, n)` 得到拓扑序
  4. 每个 agent `i` 的入边集合 = `[u for (u,v) in edges if v == i]`，按拓扑位置排序
- Routing 用**上一轮**的 contributions 算（`:534-540`），所以 round 1 的图来自 round 0 的 PC1，依次类推。

**未变**：vote 仍然等权，无早停。peer 仍然走 `last_round_responses = [ctx[-1]["content"] ...]` 快照（`:528`）—— **保持并发广播语义**，只是每个 agent 收到的 peer 子集从"4 个"缩成"≤2 个"。

**结果**：3 轮跑满，每轮每个 agent 只看到贡献分高于自己的 top-k 个 peer（被 dagify 过滤后）。

### A3_aggregation：仅打开加权投票

差分（vs A0）：

- `need_spectral = True`（需要 contributions 给投票打分）。
- `_vote` 走 `enable_weighted_aggregation=True` 分支（`:283-285`）：`max(groups, key=lambda g: (g["size"], g["weight"]))`。
  - **关键性质**：size 永远是主排序键，contributions 只是 size 平票时的 tiebreaker。所以"3 票的组"永远盖过"2 票但贡献分高的组"。

**未变**：无早停，peer 全连接 mesh。

**结果**：3 轮跑满，最终票只在 size 平票时由 contributions 决定。在 5 个 agent 上能形成 size 平票的场景其实有限（2-2-1、2-1-1-1 等），所以这个模块的边际效应理论上偏小。

### A4_all：三件套全开

= A1 + A2 + A3 的并集，没有新代码路径。这是 paper 的 headline cell。

**未变**：基础条件、prompt、role、judge、shared pool、agents/rounds、equivalence 协议 —— 全部和 A0 一致。

### A5_trig_rout：A4 减掉加权聚合

= A1 + A2，但 `enable_weighted_aggregation=false`。

**设计意图**：A4 − A5 控制了 Triggering + Routing 后，剩余差异就是 contribution-weighted aggregation 的纯效应。如果 A4 − A5 接近 0，就证明 weighted aggregation 在这个 baseline 上没贡献，这是论文 ablation 表里的关键断言之一。

## B. soo_scc 内部演化（A0 → A5）

### 跨 A0–A5 严格相等的基础条件

| 项 | 值 | 来源 |
|---|---|---|
| 父类 | `SOO_Centered_v3_Main` → `SOO_Centered_v2_Main` → `SelfOrg_Main` | `soo_scc_main.py:85` |
| Round-0 prompt | `_init_prompt` 三档（math+few-shot / mcq+format hint / open） | `soo_centered_v3_main.py:180-207` |
| Round-update prompt | `_update_prompt` 三档 | `:209-235` |
| Leader-agent prompt | `_update_prompt_leader_agent` 三档 | `:237-257` |
| 6 个 agent role | `["Assistant", "Programmer", "Mathematician", "Economist", "Psychologist", "Historian"]` | yaml |
| Few-shot 配置 | `include_math_few_shot=true, math_mode=complex, include_mcq_format_hint=true` | yaml（**6 个变体全一致**） |
| `num_agents` / `max_rounds` | 5 / 3 | yaml |
| Round-0 答案 | 从 `POOL_DIR` 回放（同一份缓存） | `run_soo_scc_1.5b.sh:60,156-157` |
| `force_task_type` | `null`（同一套 `_detect_task_type` 启发式判定） | yaml |
| Judge | xverify-9b-c，cached on canonical | yaml |
| `random_seed` | 2025 | yaml |
| Embedding 模型 | `model/all-MiniLM-L6-v2` | yaml |
| `top_k` / `sim_threshold` / `diversity_p` / `enforce_dag` | 2 / 0.75 / 0.0 / true（仅 routing 开时生效） | yaml |
| `variance_consensus_thr` / `answer_consensus_min_*` | 0.05 / 3 / 3 | yaml |
| `aggregate_mode` / `reform` / `consensus_min_sim` / `consensus_range_eps` | weighted / true / 0.95 / 0.1 | yaml |
| `temperature` | 0.5 | yaml |
| `emit_diagnostic` | true | yaml |

### 仅有的差异：四个 `enable_*` 开关

注意 soo 侧把 mad_scc 那一个"Triggering"开关拆成了两个：`enable_spectral_consensus`（tr(S_c) 闸门）和 `enable_answer_consensus`（答案聚类 ≥3 闸门）。但在 A1/A4/A5 里这两个总是**同步开**，所以叙事上仍是"一个 Triggering 开关"。

| Variant | `enable_spectral_consensus` | `enable_answer_consensus` | `enable_contribution_routing` | `enable_contribution_aggregation` |
|---|---|---|---|---|
| A0_vanilla | false | false | false | false |
| A1_triggering | **true** | **true** | false | false |
| A2_routing | false | false | **true** | false |
| A3_aggregation | false | false | false | **true** |
| A4_all | true | true | true | true |
| A5_trig_rout | true | true | true | false |

### A0_vanilla：行为锚点

代码路径（`soo_scc_main.py:149-268` 默认快速路径 + 继承自 v3 的 `_inference_with_diagnostic` 用于 emit）：

1. **Round 0**：回放 shared pool（`:184-198`），跑 `_init_prompt` 任务自适应 prompt（**仍带 few-shot 和 format hint**）。
2. **Spectral 分析**：`pc1_contributions(embs)` **无条件计算**（`:201-202`）—— A0 也会算，因为这套基线把它当成 default。
3. **早停闸门 A**：`if self.enable_answer_consensus and canonical and size >= 3` → **A0 这里短路，因为 `enable_answer_consensus=false`**。
4. **早停闸门 B**：`if self.enable_spectral_consensus and is_spectral_consensus(...)` → A0 短路。
5. **DAG 构图**：`build_diverse_graph(sims, contributions, n, ..., enable_routing=false, ...)`。`enable_routing=false` 让 `build_diverse_graph` 走 fallback 分支（`scc_components/routing.py`，等价于 `soo_centered_v3_main.py:392-397`）：**先生成全连接 `{(j,i)|j≠i}`、再 dagify**。
6. **DAG 拓扑序**：`topo_order_by_contributions(edges, contributions, n)` —— **即使 routing 关掉，拓扑序仍按 contributions 排序**（高贡献先说）。
7. **级联传播**：`_propagate_with_typed_prompts` 按拓扑序逐个 agent 调 LLM；后说的 agent 看到当轮前面 agent **已更新**的答案。
8. **mid-debate 早停**：闸门 A/B 都被关掉，所以 3 轮跑满。
9. **最终聚合**：`count_first_plurality(final_answers, contributions, task_type, enable_contribution_aggregation=false)`（`:254-257`）—— score 退化为 `(size, -groups.index(g))`，contributions 不参与。
10. **抽取失败 fallback**：`if not self.enable_contribution_aggregation: return {"response": init_answers[0]}`（`:262-263`）—— 直接返回第一个 agent 的初始答案，不做 weighted centroid。

**结果**：相对 v3 默认版本，A0 关掉的是早停 + 贡献权重；保留的是任务自适应 prompt + 任务感知聚类 + DAG-cascade 串行级联 + 贡献分驱动的拓扑序。

### A1_triggering：打开两个早停闸门

差分（vs A0）：

- `enable_answer_consensus=true` → round 0 后立刻检查 `count_first_plurality` 命中 ≥3 → 直接 `return {"response": format_final(canonical, task_type)}`（`:210-215`）
- `enable_spectral_consensus=true` → 检查 `is_spectral_consensus(spec, 0.05)` → `return _aggregate_from(...)`（`:218-225`）
- mid-debate 同样的两道闸门在每轮结束后再查（`:326-341`）

**未变**：DAG 仍然 full-mesh-dagified（routing 关），聚合仍然 count-only（aggregation 关）。

**结果**：可能在 round 0 直接结束（连 debate 都不进），可能 round 1/2 早停，也可能跑满 3 轮。

### A2_routing：打开 contribution-DAG

差分（vs A0）：

- `enable_contribution_routing=true` → `build_diverse_graph` 走主路径（`scc_components/routing.py`），按 `sim*(1+N*(c_j-c_i))` 打分、`sim_threshold=0.75` 剪枝、`top_k=2`、`diversity_p=0.0`（无随机扰动）、dagify
- 入边集合从"全连接 dagify 后剩下的所有 predecessor"缩成"top-2 高贡献 predecessor"

**未变**：早停关，聚合 count-only。3 轮跑满。

### A3_aggregation：打开贡献加权聚合

差分（vs A0）：

- `enable_contribution_aggregation=true` → `count_first_plurality` 改用 `(size, sum_contributions)` tiebreak（仍然 count-first lex）
- `_aggregate_from` 用 weighted centroid + 最近邻 answer 作为 extraction-failure fallback（而不是 A0 的"返回第一个 agent 初始答案"）

**未变**：早停关，routing 关（仍然 full-mesh-dagified）。3 轮跑满。

### A4_all：四件套全开

= A1 + A2 + A3 的并集。等价于 v3 默认 `config_main.yaml`，只是把 `diversity_p` 强制设成 0（关掉随机扰动）以保证可复现。

### A5_trig_rout：A4 减掉加权聚合

= A1 + A2，`enable_contribution_aggregation=false`。

**设计意图**：与 mad_scc A5 同构。**A4 − A5 = contribution-weighted aggregation 在这条 baseline 上的纯效应**。

## 结论

**在每个方法内部，A0–A5 是严格控制变量的设计**：

1. 所有 prompt 模板（包括 few-shot）、role personas、judge、agent 数、轮数、随机种子、shared initial pool、equivalence 协议、embedding 模型 —— 这些"评判基础条件"在 6 个变体里**逐字段相等**。
2. yaml diff 只有 3 行（mad_scc）或 4 行（soo_scc）—— 全部是 `enable_*` 布尔开关。
3. **Round-0 答案被强制共享**（`POOL_DIR` 机制），所以 6 个变体的初始诊断桶分母完全相同。`analyze_diagnostic.py --strict` 会硬验证这点。
4. 任务自适应 prompt 是 **method 内部的常量**，不是 ablation 维度。它只在 mad_scc vs soo_scc **跨方法比较**时才成为混淆因素。

所以论文里可以放心写：

> Within each base method, A0–A5 differ **only** in the activated SCC modules; all other inference conditions (prompts, role personas, judge model, agent count, round budget, embedding encoder, equivalence protocol, and the round-0 answer pool) are held identical across variants via a shared initial-pool replay mechanism.

跨簇（mad_scc 的 A4 vs soo_scc 的 A4）则**不能**说成同样话 —— 那是另一回事。
